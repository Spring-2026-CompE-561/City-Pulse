Part 5: Additional Considerations

Authentication:
	For authentication, we are using FastAPI OAuth2PasswordBearer with tokenUrl="api/auth/login" (auto_error=False). Users log in via POST /api/auth/login with email and password; the API returns an access_token to be sent as Authorization: Bearer <token>. We use an in-memory token store (token -> user_id, expires_at) with configurable expiry; tokens are invalid on server restart. Passwords are hashed with SHA-256. We provide get_current_user (optional: returns User or None) and get_current_user_required (returns User or 401) as dependencies for protecting routes.

Middleware:
	For middleware, we are not using any custom middleware at this time (no CORS, logging, or other middleware registered in the FastAPI app). All request/response handling is done in route handlers and dependencies.

Error Handling:
	For error handling, we are using a global exception handler in main.py that catches unhandled Exception, re-raises HTTPException so FastAPI can return the intended status and body, and for other exceptions returns a consistent JSONResponse with status 500 and body {"detail": "...", "success": false}. In debug mode the detail includes the exception type and message. Per-route we raise HTTPException with appropriate status codes (400, 401, 403, 404, 409) and detail messages across auth, users, regions, events, trends, and interactions routers.

Testing:
	For testing, we are configured to use pytest with pytest-asyncio (asyncio_mode = "auto") and httpx for API tests; testpaths is set to src/test in pyproject.toml. The testing stack is in place; test modules can be added under src/test as needed.
